# Templating Reference

Agent prompt files and `CLAUDE.md` are processed as templates at install time. This lets a single source file adapt to whichever agents the user selected, which model they are targeting, and any project-level settings without duplicating content.

Template processing runs over every agent file and the team's `CLAUDE.md` immediately after download, before anything is written to the target project.

---

## Block Tags

Block tags conditionally include or exclude entire sections of a file.

### Agent presence

```
<%if agent "agent-id"%>
  This block is only included when "agent-id" was selected for installation.
<%endif%>

<%if not agent "agent-id"%>
  This block is only included when "agent-id" was NOT selected.
<%endif%>
```

**Example:** the designer handoff step in `CLAUDE.md` should only appear when the Principal Designer is installed:

```
<%if agent "principal-designer"%>

**Step 1.1:** The **principal-designer** is consulted by the requirements engineer.
Both agents discuss the task and the designer advises on UX direction.

<%endif%>
```

### Settings condition

```
<%if settings.key == value%>
  Included when the settings map contains key=value.
<%endif%>

<%if not settings.key == value%>
  Included when it does not.
<%endif%>
```

Settings are arbitrary key/value pairs passed in from the CLI installer (e.g. `llmTarget`, feature flags). Values are always compared as strings.

## Inline Tags

Inline tags are substituted within a line of text.

### Role resolution

```
<%agent role%>
```

Resolves to the **ID of the agent that holds that role** in the current installation. This lets `CLAUDE.md` and agent prompts reference each other by role rather than by hardcoded ID, so teams can swap out an agent without updating every cross-reference.

**Example:**

```
When architecture guidance is needed → invoke **<%agent architect%>**
```

After processing with the Swift Apple team selected, this becomes:

```
When architecture guidance is needed → invoke **swift-app-architect**
```

Roles are declared in the team YAML under each agent entry:

```
architects:
  - id: swift-app-architect
    file: Teams/swift-apple/swift-app-architect.md
    role: architect        # ← this is what <%agent architect%> resolves to
```

If no agent holds the requested role, the tag is left as-is (no silent failure).

### Variable substitution

```
<%varname%>
```

Substitutes any key from the variables map. Currently the installer exposes one built-in variable:

Variable
Value

`<%model%>`
The model string for this agent (`opus`, `sonnet`, `gpt-4o`, etc.), taken from the agent's `models` entry in the team YAML

**Example** — in an agent's frontmatter:

```
---
model: <%model%>
---
```

After processing this becomes:

```
---
model: opus
---
```

Unknown tags (no matching key in the variables map) are left untouched.

### Roster injection

```
<%roster%>
```

Replaced with the full contents of `AGENT_ROSTER.md`, which is auto-generated during installation. The roster lists every installed agent's ID, role, file path, and description summary.

This is useful in orchestrator prompts that need to know the full team composition at runtime.

---

## How Templates Are Processed

The install sequence is:

1. All agent files for the selected team are downloaded raw.
2. `AGENT_ROSTER.md` is generated by reading frontmatter from the downloaded files.
3. Each agent file is processed through the template engine and written to `.claude/agents/`.
4. `CLAUDE.md` is fetched (from the team's `claude_md` path), processed through the same engine, and written to `.claude/CLAUDE.md`.

The template engine receives:

- `selectedAgentIds` — the list of agent IDs the user chose
- `rosterContent` — the generated roster markdown
- `settings` — key/value pairs from the CLI (e.g. `{ llmTarget: 'claude' }`)
- `roleMap` — a map of `role → agent-id` built from the selected agents
- `variables` — per-agent values like `model`

## Writing a New Team

When building a new team, use block tags to make your `CLAUDE.md` adapt to optional agents:

```
## Workflow

**Step 1:** <%agent requirements%> → define the feature spec

<%if agent "principal-designer"%>
**Step 1.1:** <%agent designer%> → UX direction
<%endif%>

**Step 2:** <%agent product-owner%> → break spec into tasks
**Step 3:** <%agent architect%> → add structural guidance
**Step 4:** <%agent developer%> → implement
```

And use `<%model%>` in agent frontmatter so model selection flows through from the YAML:

```
---
name: My Engineer
description: Implements features for my platform
model: <%model%>
---
```

See [Teams/swift-apple/](../Teams/swift-apple/) for a complete working example.
